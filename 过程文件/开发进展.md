# 币安合约横盘监控 — 开发进展

## 核心目标
监控全网 Binance USDT 永续合约中长期横盘、具备爆发潜力的标的，按横盘持续时间降序排列。

## 当前状态：✅ 架构拆分完成，核心引擎已在云端验证通过

---

## 架构设计

### 分层架构图
```
┌─────────────────────────────────────────────────┐
│  .github/actions/setup-tor/action.yml           │  ← Tor 初始化 (Composite Action, 可复用)
└──────────────────────┬──────────────────────────┘
                       │ uses: ./.github/actions/setup-tor
┌──────────────────────┴──────────────────────────┐
│  binance_gateway.py                              │  ← 通用网关 (Session/重试/API 封装)
│  ├── create_session()                            │
│  ├── fetch_json(url, retries, timeout)           │
│  ├── get_all_usdt_perpetuals()                   │
│  ├── fetch_klines(symbol, interval, limit)       │
│  ├── fetch_funding_rate()        [预留]           │
│  └── fetch_open_interest()       [预留]           │
└──────────────────────┬──────────────────────────┘
                       │ from binance_gateway import ...
     ┌─────────────────┼─────────────────┐
     ▼                 ▼                 ▼
 sideways_scanner  (未来)持仓监控    (未来)资金费率
```

### 新增功能时只需：
1. 写一个新的 `xxx_scanner.py`，`from binance_gateway import ...`
2. 复制一份简短的 Workflow YAML，引用 `setup-tor`
3. 网络层零成本复用

## 进展日志

### 第一阶段：调研与方案选型 ✅
*   Coinglass 页面无法满足需求（无横盘时间维度、无交易所过滤）。

### 第二阶段：网络攻坚 ✅
*   公共反代节点 90%+ 不可用。
*   Cloudflare Worker 反代 (`binance.794988.xyz`) 被 451 封堵。
*   **最终方案：Tor 匿名网络**，强制 9 国出口节点。

### 第三阶段：引擎构建 ✅
*   向后追溯法横盘判定算法。
*   20 线程并发，544 合约全扫约 10 秒。

### 第四阶段：云端部署 ✅
*   GitHub Actions 定时巡检（每日 12:00 / 20:00 北京时间）。
*   报告直出日志 + 可下载 Artifact。

### 第五阶段：架构拆分 ✅ (2026-03-07)
*   抽取 `binance_gateway.py` 通用网关模块。
*   抽取 `.github/actions/setup-tor/` Composite Action。
*   `sideways_scanner.py` 瘦身为纯业务逻辑。

## 关键配置参数
| 参数 | 当前值 | 含义 |
|---|---|---|
| `INTERVAL` | `1h` | K 线级别 |
| `LIMIT` | `200` | 最大追溯 K 线数 |
| `AMPLITUDE_THRESHOLD` | `0.02` (2%) | 横盘振幅上限 |
| 最低横盘门槛 | `> 6 小时` | 报告过滤阈值 |
| `BBW_THRESHOLD` | `0.05` (5%) | 布林带宽度阈值 (判断收敛的标准) |
| `BB_TOLERANCE` | `1` | 允许突发插针打断收敛的容错根数 |
| `MIN_DURATION` | `6` | 报告过滤门槛 (收敛必须 > 6 根 K 线) |
